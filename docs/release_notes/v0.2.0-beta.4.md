# v0.2.0-beta.4

这次 beta 聚焦 TIFF 导出可靠性、Windows 11 可用性和 release 验证流程。
重点修复了 Windows 上非 ACEScg 色彩空间导出明显变慢、看起来像卡死的
问题，同时明确废除 LUT 近似导出路径，保持最终写入 TIFF 的 code value
来自精确的色彩空间矩阵、transfer function 和量化步骤。

这仍然是 beta 版本。它适合继续测试 trichrome 导入、预览、16-bit TIFF
导出、ICC profile、已命名目标色彩空间和 RAW 工作流边界，但不代表已经
完整覆盖所有相机 RAW，也不声明 RAW 色彩正确性已经达到 RawTherapee
parity。

## Changes

- Fixed the export hot path that made non-ACEScg color spaces much slower on
  Windows CPU fallback. The RGB color-space conversion matrix is now computed
  once per exported image instead of once per pixel.
- Removed the LUT export approach from the final implementation. Export keeps
  exact transfer-function evaluation and final integer quantization, matching
  the staged-writer contract used by SoftLoaf Negative.
- Added export timing logs around CPU QImage encoding so future reports can
  separate decode/compose, encode, TIFF write, and total export time.
- Confirmed that the current Windows D3D12 export backend is still a placeholder
  and falls back to CPU. The beta.4 fix is therefore a CPU fallback fix, not a
  partial GPU path.
- Added fixed Windows 11 `303mini` environment notes under
  `docs/windows11_303mini_environment.md` and pointed `AGENTS.md` at them so
  future Windows validation reuses the known MSVC/Qt/vcpkg setup.

## Known Limitations

- Windows export encoding still uses CPU fallback until the D3D12 backend is
  implemented.
- macOS CLI/probe validation in this cycle also reported CPU fallback for the
  export backend; do not treat the measured beta.4 numbers as Metal throughput.
- DNG import remains LibRaw-backed; DNG export is not implemented in this
  release.
- RAW color correctness still does not claim RawTherapee parity. Remaining RAW
  compatibility and color boundaries are tracked in `docs/raw_correctness_report.md`.
- Windows packages may trigger SmartScreen when Authenticode signing is not
  configured.

## Verification

Local macOS validation:

```text
cmake --build build --target softloaf_trichrome_app trichrome_smoke
./build/trichrome_desktop_core_test
./build/trichrome_smoke
ctest --test-dir build --output-on-failure
```

Windows 11 `303mini` validation:

```text
cmake --build build-windows --target softloaf_trichrome_app trichrome_smoke trichrome_desktop_core_test raw_fixture_regression_test --config Release
ctest --output-on-failure -C Release
```

Actual 16-bit TIFF export timing was measured with the July 9 trichrome sample:

```text
macOS source: /Volumes/T7 Touch/Film/2026-07-09
Windows source: C:\Users\flemyng.303mini\Desktop\film\20260709
image size: 6464 x 4848
output size: about 188 MB per TIFF
```

Observed Windows total export time was about `1.1s` for ACEScg and
`1.8s-1.9s` for non-ACEScg named color spaces on CPU fallback. Observed macOS
CPU fallback total export time was about `5.3s` for ACEScg and `6.3s-6.4s` for
non-ACEScg named color spaces from the external T7 source volume.

## Packaging Notes

- macOS DMG packaging must pass bundle dependency resolution, code signing,
  notarization, and macOS 15.0 `minos` compatibility checks.
- Windows release publishes both installer and portable zip packages with Qt,
  OpenCV, LibRaw, Little CMS, and runtime dependencies.
- Windows release artifacts remain unsigned unless the workflow has
  Authenticode credentials configured.
